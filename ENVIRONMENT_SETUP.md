# 🔐 Environment & Deployment Security Setup

## Overview

This document explains how environment variables, RDS, and secrets are configured for this serverless application.

**Key Principle:** Secrets are never stored in version control. They're managed through:

- 🔒 SST Secrets Manager (for deployed stages)
- 📝 .env.local file (for local development, git-ignored)

## Deployment Setup

### Stage: Development (`dev`)

**RDS Configuration:**

- SST creates managed RDS instance + RDS Proxy automatically
- Private VPC networking (no public DB access)
- Optional external DB override available via `DatabaseUrl`

**Step-by-step:**

```bash
# 1. Set required application and DB secrets
sst secret set -s dev StripeSecretKey "sk_test_your_test_key"
sst secret set -s dev StripeWebhookSecret "whsec_your_webhook_secret"
sst secret set -s dev BookingChargeKey "your_development_charge_key"
sst secret set -s dev DatabasePassword "development_password"

# 2. Optional: use external RDS instead of SST-managed RDS
sst secret set -s dev DatabaseUrl "postgresql://postgres:password@your-rds-host:5432/vertiaccess?sslmode=require"
USE_EXTERNAL_DATABASE=true sst deploy -s dev

# 3. Default deploy (managed RDS)
sst deploy -s dev

# 4. View outputs
sst output -s dev
```

### Stage: Staging (`staging`)

**RDS Configuration:**

- SST creates managed RDS instance
- Not publicly accessible (private VPC only)

**Step-by-step:**

```bash
# 1. Set production-grade secrets
sst secret set -s staging StripeSecretKey "sk_test_staging_key"
sst secret set -s staging StripeWebhookSecret "whsec_staging_webhook"
sst secret set -s staging BookingChargeKey "staging_charge_key"
sst secret set -s staging DatabasePassword "staging_secure_password"

# 2. Deploy
sst deploy -s staging

# 3. Run integration tests
# Test all endpoints, Stripe webhooks, payments, etc.
```

### Stage: Production (`production`)

**RDS Configuration:**

- SST creates managed RDS instance
- Database configured for High Availability
- Automatic backups enabled
- Private VPC, no public access

**CRITICAL STEP - Set Production Secrets:**

```bash
# ⚠️ DO THIS FIRST - BEFORE DEPLOYMENT

# 1. Stripe Production Keys (get from Stripe account)
sst secret set -s production StripeSecretKey "sk_live_your_production_key"
sst secret set -s production StripeWebhookSecret "whsec_prod_live_webhook_secret"

# 2. Booking charge key
sst secret set -s production BookingChargeKey "your_production_charge_key"

# 3. Database password (STRONG - 16+ chars, mixed case, numbers, symbols)
sst secret set -s production DatabasePassword "Pr0d!Secure#Pass2026@VertiaAccess"

# 4. Verify secrets are set
sst secret list -s production

# 5. NOW Deploy
sst deploy -s production
```

## RDS Details

### Database Instance Specifications

**Development Stage (`dev`):**

- Uses SST-managed RDS + RDS Proxy by default
- Optional external DB via `DatabaseUrl` with `USE_EXTERNAL_DATABASE=true`
- Auto-backup every 24 hours
- No downtime during updates

**Staging/Production Stages:**

- Instance Type: `db.t4g.micro`
- Engine: PostgreSQL
- Storage: 20 GB (auto-scaling available)
- Multi-AZ: Not enabled for cost (enable for production)
- Backup Retention: 7 days
- No public access (VPC only)

### Connection Flow

```
Lambda Function
    ↓
    ├─→ RDS Proxy (connection pooling)
    │
    └─→ RDS Instance (PostgreSQL)
            ↓
        vertiaccess database
```

### Password Management

- ✅ Never stored in code
- ✅ Managed by AWS Secrets Manager
- ✅ Automatic rotation available
- ✅ Accessed via IAM roles

**To reset password:**

```bash
sst secret set -s production DatabasePassword "NewSecurePassword123!"
```

## CORS Configuration for Frontend

### Automatic CORS Handling

The `sst.config.ts` automatically configures CORS to include:

1. ✅ Local development URLs: `http://localhost:3000`, `http://localhost:5173`
2. ✅ Deployed frontend URL: e.g., `https://your-app.cloudfront.net`
3. ✅ Any custom origins from `.env` `ALLOWED_ORIGINS`

**How it works:**

```typescript
// In sst.config.ts
const corsOrigins = (() => {
    const origins = [
        'http://localhost:3000', // Local frontend
        'http://localhost:5173', // Vite dev
        frontend.url, // CloudFront URL (auto-added)
    ];
    const envOrigins = process.env.ALLOWED_ORIGINS?.split(',') || [];
    return [...new Set([...origins, ...envOrigins])].join(',');
})();
```

### Verifying CORS Works

```bash
# Test CORS from terminal
curl -i -X OPTIONS https://your-api-url.com/auth/v1/health \
  -H "Origin: https://your-frontend-url.com" \
  -H "Access-Control-Request-Method: GET"

# Response should include:
# Access-Control-Allow-Origin: https://your-frontend-url.com
```

### Custom CORS Origins

If you need additional origins:

```env
# In .env.local
ALLOWED_ORIGINS=http://localhost:3000,http://localhost:5173,https://custom-domain.com
```

## Stripe Webhook Configuration

### After Deployment

Once deployed, configure webhooks in Stripe Dashboard:

**Steps:**

1. Go to: https://dashboard.stripe.com/webhooks
2. Click "Add Endpoint"
3. Endpoint URL: `https://your-deployed-api.com/billing/v1/webhooks/stripe`
4. Events to send:
    - `payment_intent.succeeded`
    - `payment_intent.payment_failed`
    - `customer.subscription.updated`
    - `customer.subscription.deleted`
5. Click "Add Endpoint"
6. Copy your signing secret
7. Set the secret in SST:
    ```bash
    sst secret set -s production StripeWebhookSecret "whsec_your_webhook_secret"
    ```

## Security Best Practices

### DO ✅

- ✅ Use SST secrets for all sensitive data
- ✅ Use `.env.local` for local development only
- ✅ Git-ignore all `.env.*` files (already done)
- ✅ Rotate database passwords quarterly
- ✅ Use strong passwords (16+ chars, mixed case, numbers, symbols)
- ✅ Enable MFA on AWS account
- ✅ Use IAM roles (not access keys) for Lambda functions

### DON'T ❌

- ❌ Commit `.env` files to git
- ❌ Share credentials in Slack/email
- ❌ Store passwords in comments
- ❌ Use same password for all stages
- ❌ Disable SSL on database connections
- ❌ Use admin credentials in application code

## Troubleshooting

### Database Connection Fails

```bash
# 1. Check database URL
echo $DATABASE_URL

# 2. Test connection
psql $DATABASE_URL

# 3. For RDS proxy issues, check security group:
aws ec2 describe-security-groups --group-ids sg-xxx

# 4. Check RDS credentials in secrets:
sst secret list -s production
```

### CORS Errors in Frontend

```bash
# Check what origins are configured
sst output -s production

# The apiUrl and frontendUrl should match
# If not, redeploy:
sst deploy -s production
```

### Secrets Not Being Used

```bash
# List all secrets
sst secret list -s production

# Check Lambda environment variables
aws lambda get-function-configuration --function-name vertiaccess-production-AuthService

# Redeploy if secrets were just set
sst deploy -s production
```

### Can't Deploy - Permission Errors

```bash
# Check AWS credentials
aws sts get-caller-identity

# Verify IAM permissions
aws iam get-user

# Configure credentials
aws configure
```
